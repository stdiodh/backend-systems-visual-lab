# 01. 들어가며

## 1. 이 시퀀스의 역할

이 시퀀스는 사용자가 “코딩을 할 수 있게 된 것”과 “실무 백엔드 시스템을 판단할 수 있게 된 것”의 차이를 이해하도록 만든다. Visual Lab의 사용법, 학습 흐름, Codex 작업 규칙, 시나리오 기반 학습 방식을 고정한다.

**이전 시퀀스와의 연결**  
없음. 전체 랩의 진입점이다.

**다음 시퀀스로 넘길 질문**  
단순히 코드가 동작하는 서비스가 사용자가 늘어나면 왜 느려지는가?

**검색/데이터 별칭**  
intro, onboarding, lab-guide

## 2. 시작 Incident

새로운 개발자가 CRUD API를 만들 수 있게 되었지만, 서비스가 느려지거나 장애가 발생했을 때 어디부터 봐야 하는지 모른다. 기능 구현은 가능하지만 운영 상황에서 원인을 좁히는 기준이 없다.

### 사용자가 처음 봐야 하는 질문

“코드가 돌아간다”는 사실만으로 서비스를 운영할 수 있을까?

## 3. 세부 목차

- 코딩을 할 수 있게 된 것일 뿐
- Visual Lab을 읽는 방법
- 문제 상황 기반 학습 방식
- Codex가 참조해야 하는 문서 순서
- 장별 학습 흐름과 부록의 역할

## 4. 관찰 지표

- 기능 구현 중심의 사고에서 시스템 관찰 중심 사고로 전환했는가
- 각 장이 독립 카드가 아니라 하나의 장애 흐름으로 연결되는가
- Codex가 curriculum-map만 보고 구현하지 않도록 문서 참조 순서가 고정되었는가

## 5. 원인 후보

- 이론을 목차 순서로만 나열하면 사용자가 실무 상황과 연결하지 못한다
- UI가 예쁘기만 하면 핵심 판단 기준이 사라진다
- Codex가 문서 참조 규칙 없이 사이트를 만들면 카드형 대시보드로 막 생성될 수 있다

## 6. 핵심 개념

### Incident-driven Learning

**정의**  
개념을 먼저 설명하지 않고 장애 상황이나 운영 문제에서 출발하는 학습 방식이다.

**실무에서 중요한 이유**  
실무 백엔드 문제는 대부분 증상에서 시작해 원인을 좁히는 과정으로 해결되기 때문이다.

**흔한 오해**  
개념 설명을 많이 넣으면 좋은 학습이라고 착각하기 쉽다. 하지만 사용자가 먼저 봐야 하는 것은 긴 설명이 아니라 “무슨 문제가 생겼는가”이다.

**Visual Lab 반영 방식**  
첫 화면에 “서비스가 느려졌다”, “외부 API가 멈췄다”, “데이터가 꼬였다” 같은 Incident를 배치한다.

**Codex 누락 방지 규칙**  
각 시퀀스 문서는 반드시 시작 Incident를 가져야 하며, 단순 정의로 시작하지 않는다.

### System Thinking

**정의**  
API, DB, 외부 API, 메시지 큐, OS, 네트워크, 보안, 아키텍처를 하나의 연결된 시스템으로 보는 사고 방식이다.

**실무에서 중요한 이유**  
현상 하나가 여러 계층에서 원인을 가질 수 있기 때문에 특정 기술만 보면 장애를 오진하기 쉽다.

**흔한 오해**  
Spring Boot 코드만 잘 작성하면 서버 문제를 해결할 수 있다고 생각하는 것이다.

**Visual Lab 반영 방식**  
System Map Panel에서 요청이 어느 계층을 지나고 어떤 리소스를 사용하는지 시각화한다.

**Codex 누락 방지 규칙**  
모든 장은 적어도 하나 이상의 상위 시스템 맵과 연결되어야 한다.

### Codex Guardrail

**정의**  
Codex가 구현 전 반드시 읽어야 하는 문서, 만들면 안 되는 것, 수정 범위를 제한하는 규칙이다.

**실무에서 중요한 이유**  
문서 없이 UI부터 만들면 이론 누락, 디자인 혼선, 범위 폭주가 발생한다.

**흔한 오해**  
프롬프트가 자세하면 자동으로 좋은 결과가 나온다고 생각하는 것이다. 프롬프트보다 기준 문서와 체크리스트가 더 중요하다.

**Visual Lab 반영 방식**  
화면에는 직접 노출하지 않지만 `docs/prompts`, `AGENTS.md`, `docs/sequences`가 구현 전 참조 기준이 된다.

**Codex 누락 방지 규칙**  
Phase 2 이후 모든 작업 프롬프트는 대상 sequence 문서를 읽으라는 문장을 포함해야 한다.

### Visual Lab

**정의**  
정적 문서가 아니라 문제 흐름, 지표, 선택지, 시스템 흐름을 시각적으로 따라가는 학습 화면이다.

**실무에서 중요한 이유**  
백엔드 실무 개념은 글만으로 이해하기 어렵고 상태 변화, 병목 위치, 요청 흐름을 봐야 이해가 빨라진다.

**흔한 오해**  
예쁜 랜딩 페이지나 일반 문서 사이트를 Visual Lab으로 착각하는 것이다.

**Visual Lab 반영 방식**  
Incident Panel, Metric Card, Flow Explorer, Decision Panel, Concept Card를 조합한다.

**Codex 누락 방지 규칙**  
긴 본문을 그대로 HTML에 넣지 말고 화면 요소별로 분해한다.


## 7. 실무 판단 기준

- 이 랩은 문서 사이트가 아니라 실무 판단 훈련용 Visual Lab으로 구현한다
- 각 장은 “문제 상황 → 관찰 → 원인 후보 → 선택지 → 다음 질문”을 따른다
- 정의 사전과 시퀀스 문서를 먼저 만들고 그다음 디자인을 입힌다
- 초기 구현은 모든 장의 완성보다 일관된 UI 패턴과 01~03장 품질을 우선한다

## 8. Visual Lab 화면 반영 방식

### 필수 컴포넌트

- `Hero Section`
- `Codex Terminal Panel`
- `Chapter Progress Rail`
- `System Map Panel`
- `Next Question`

### 권장 사용자 흐름

1. Hero에서 전체 랩의 목적을 확인한다
2. Codex Terminal Panel에서 구현 전 읽을 문서 순서를 보여준다
3. Chapter Progress Rail에서 01~부록 C의 전체 흐름을 본다
4. System Map Panel에서 이후 장들이 어떤 계층을 다루는지 미리 본다
5. Next Question으로 02장 성능 진단으로 넘어간다

### UX Writer 규칙

- 제목은 개념명이 아니라 문제 상황으로 시작한다.
- 첫 문장은 사용자가 지금 겪는 증상으로 쓴다.
- 해결책은 하나만 제시하지 말고 선택지를 비교한다.
- “항상”, “무조건” 같은 표현을 피하고 조건을 함께 보여준다.
- 경고 메시지는 원인을 비난하지 말고 다음 관찰 지표를 안내한다.


## Visual Lab 공통 UX/UI 원칙

이 시퀀스는 긴 이론 본문을 화면에 그대로 넣지 않는다. 사용자는 먼저 **문제 상황**을 보고, 그 다음 **관찰 지표**, **원인 후보**, **선택지**, **흐름도**, **체크리스트** 순서로 판단해야 한다.

- 첫 화면에는 `Incident Panel`을 배치한다.
- 핵심 수치는 `Metric Card`로 짧게 보여준다.
- 원인 후보는 `Candidate Cause Stack` 또는 `Decision Panel`로 보여준다.
- 개념 설명은 접히는 `Concept Card`로 제공한다.
- 코드/명령/설정 예시는 `Codex Terminal Panel` 또는 `Code Lens Panel`로 제공한다.
- 다음 시퀀스로 넘어가는 질문은 화면 하단의 `Next Question`으로 고정한다.
- 모바일에서는 Chapter Rail보다 현재 시퀀스 제목, 진행률, 다음 액션이 먼저 보여야 한다.


## 9. 데이터 모델 힌트

사이트 데이터 파일에서 이 시퀀스는 다음 필드를 가진 객체로 표현한다.

- `sequenceId`: 01
- `title`: 들어가며
- `chapterCode`: INIT
- `status`: planned | active | done
- `nextQuestion`: 02장으로 넘길 질문
- `requiredDocs`: 구현 전 읽어야 할 문서 배열

## 10. 인터랙티브 시나리오 제안

- **온보딩 모드**: 사용자가 Visual Lab을 처음 열면 전체 장 목록, 학습 방식, 구현 기준 문서를 순서대로 확인한다.
- **Codex 작업 준비 모드**: Codex에게 사이트를 맡기기 전 읽어야 할 문서와 금지 사항을 체크리스트로 보여준다.
