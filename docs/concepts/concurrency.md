# Concurrency Concepts

### Race Condition

분류: 동시성

정의: 여러 실행 흐름이 같은 데이터에 접근하고 수정하는 순서에 따라 결과가 달라지는 문제다.

실무에서 중요한 이유: 재고, 포인트, 조회수, 중복 요청 처리에서 데이터가 틀어질 수 있다.

흔한 오해: 로컬에서 한 번 실행했을 때 문제가 안 보이면 안전하다고 생각한다.

Visual Lab 반영 방식: Concurrent Request Timeline으로 두 요청의 읽기, 수정, 저장 순서를 나란히 보여준다.

Codex 누락 방지 규칙: 동시성 장에는 같은 데이터를 동시에 수정하는 예시를 반드시 넣는다.

### Lost Update

분류: 동시성 / DB

정의: 두 요청이 같은 값을 읽고 각각 수정한 뒤 나중 저장된 값이 먼저 저장된 변경을 덮어쓰는 문제다.

실무에서 중요한 이유: 재고 차감, 포인트 사용, 쿠폰 발급 같은 핵심 데이터가 틀어진다.

흔한 오해: 트랜잭션을 쓰면 lost update가 항상 막힌다고 생각한다.

Visual Lab 반영 방식: Lost Update Simulator에서 read-modify-write 순서를 단계별로 보여준다.

Codex 누락 방지 규칙: 트랜잭션 격리 수준, 버전 체크, lock 선택지를 함께 보여준다.

### 낙관적 락

분류: DB 동시성

정의: 충돌이 자주 일어나지 않는다고 보고 저장 시점에 버전 등을 확인해 충돌을 감지하는 방식이다.

실무에서 중요한 이유: 불필요한 대기를 줄이면서 데이터 충돌을 감지할 수 있다.

흔한 오해: 락이라고 하면 모두 기다리는 방식이라고 생각한다.

Visual Lab 반영 방식: Version Check Timeline으로 성공/실패 요청을 보여준다.

Codex 누락 방지 규칙: 비관적 락과 비교해서 표시한다.

### 비관적 락

분류: DB 동시성

정의: 충돌 가능성이 높다고 보고 데이터를 읽거나 수정할 때 다른 트랜잭션의 접근을 제한하는 방식이다.

실무에서 중요한 이유: 강한 정합성이 필요할 때 사용할 수 있지만 대기와 데드락 위험이 있다.

흔한 오해: 비관적 락을 쓰면 모든 동시성 문제가 해결된다고 생각한다.

Visual Lab 반영 방식: Lock Wait Queue로 대기 요청을 보여준다.

Codex 누락 방지 규칙: 성능 저하와 데드락 주의 사항을 함께 넣는다.

### Unique Constraint

분류: DB 동시성 / 데이터 무결성

정의: 특정 컬럼 또는 컬럼 조합의 값이 중복되지 않도록 DB가 강제하는 제약이다.

실무에서 중요한 이유: 애플리케이션 레벨 체크가 동시에 통과해도 DB에서 마지막 중복을 막을 수 있다.

흔한 오해: 저장 전에 조회해서 없으면 insert하면 중복이 생기지 않는다고 생각한다.

Visual Lab 반영 방식: 두 요청이 동시에 중복 쿠폰 발급을 시도하고 DB unique constraint가 하나를 막는 흐름을 보여준다.

Codex 누락 방지 규칙: 중복 생성 문제에는 애플리케이션 체크와 DB 제약의 차이를 포함한다.

### Single Writer

분류: 동시성 / 처리 구조

정의: 특정 데이터 변경을 하나의 작업자 또는 순차 처리 흐름으로 모아 동시에 수정되지 않게 하는 방식이다.

실무에서 중요한 이유: 락 경쟁을 줄이고 처리 순서를 단순하게 만들 수 있다.

흔한 오해: 모든 문제는 DB 락으로만 해결해야 한다고 생각한다.

Visual Lab 반영 방식: Single Worker Queue Panel에서 여러 요청이 하나의 worker를 통해 순서대로 처리되는 모습을 보여준다.

Codex 누락 방지 규칙: 동시성 해결책에는 lock 외에 queue 기반 단일 처리 선택지도 포함한다.

### Deadlock

분류: DB 동시성 / 장애

정의: 두 개 이상의 트랜잭션이 서로 가진 잠금을 기다리며 더 이상 진행하지 못하는 상태다.

실무에서 중요한 이유: 잠금 순서와 트랜잭션 범위를 잘못 잡으면 정상 요청도 실패하거나 지연된다.

흔한 오해: 비관적 락을 쓰면 정합성은 좋아지고 부작용은 거의 없다고 생각한다.

Visual Lab 반영 방식: Lock Wait Timeline에서 트랜잭션 A/B가 서로 다른 row를 잡고 기다리는 구조를 보여준다.

Codex 누락 방지 규칙: 비관적 락 설명에는 deadlock 위험과 잠금 순서 규칙을 함께 넣는다.
