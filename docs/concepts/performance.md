# Performance Concepts

### 병목

분류: 성능 공통

정의: 전체 요청 처리 속도를 제한하는 가장 느리거나 가장 먼저 포화되는 지점이다.

실무에서 중요한 이유: 병목이 DB인지, 외부 API인지, 스레드인지, 네트워크인지 모르면 잘못된 곳을 최적화한다.

흔한 오해: 응답이 느리면 무조건 서버 코드가 문제라고 생각한다.

Visual Lab 반영 방식: Request Flow Explorer에서 각 구간의 대기 시간을 다르게 표시하고 가장 오래 걸리는 구간을 강조한다.

Codex 누락 방지 규칙: 02장 이후 성능 관련 화면에는 원인 후보와 병목 후보를 분리해서 표시한다.

### 처리량

분류: 성능 지표

정의: 단위 시간 동안 처리한 요청 수다. 보통 RPS 또는 TPS로 표현한다.

실무에서 중요한 이유: 서버가 어느 정도의 요청량을 감당하는지 판단하는 기준이다.

흔한 오해: 처리량이 높으면 응답 시간도 항상 빠르다고 생각한다.

Visual Lab 반영 방식: Metric Card에서 RPS/TPS를 응답 시간과 나란히 보여준다.

Codex 누락 방지 규칙: 처리량은 응답 시간, 에러율, 자원 사용률과 함께 보여준다.

### 응답 시간

분류: 성능 지표

정의: 요청이 들어온 순간부터 응답을 받기까지 걸린 시간이다.

실무에서 중요한 이유: 사용자가 직접 체감하는 성능이다.

흔한 오해: 평균 응답 시간만 보면 충분하다고 생각한다.

Visual Lab 반영 방식: 평균, p95, p99를 함께 보여주는 카드와 분포 시각화를 둔다.

Codex 누락 방지 규칙: 평균 응답 시간만 단독으로 쓰지 않는다.

### p95 / p99

분류: 성능 지표

정의: 전체 요청 중 95% 또는 99%가 이 시간 안에 처리됐다는 의미의 백분위 지표다.

실무에서 중요한 이유: 평균으로 숨겨지는 느린 요청을 찾을 수 있다.

흔한 오해: p99가 평균보다 큰 것이 이상한 현상이라고 생각한다.

Visual Lab 반영 방식: 응답 시간 분포 바와 tail latency 설명 카드로 표현한다.

Codex 누락 방지 규칙: 성능 장에서는 평균과 백분위 지표의 차이를 반드시 설명한다.

### 포화점

분류: 성능 테스트

정의: 요청량이 늘어도 처리량은 더 이상 크게 늘지 않고 응답 시간과 에러가 증가하기 시작하는 지점이다.

실무에서 중요한 이유: 서비스가 감당할 수 있는 한계를 찾고 증설 또는 구조 개선 시점을 판단할 수 있다.

흔한 오해: 서버가 죽는 순간만 한계라고 생각한다.

Visual Lab 반영 방식: Load Test Curve에서 처리량, latency, error rate 변화를 함께 표시한다.

Codex 누락 방지 규칙: 부록 A에는 포화점과 버클존을 숫자 하나가 아니라 곡선 변화로 보여준다.

### Latency Distribution

분류: 성능 지표

정의: 요청 응답 시간이 어떤 구간에 얼마나 분포하는지 보여주는 형태다.

실무에서 중요한 이유: 평균 하나로는 대부분 빠르지만 일부가 매우 느린 상황을 볼 수 없다.

흔한 오해: 평균 응답 시간이 낮으면 사용자 경험도 항상 좋다고 생각한다.

Visual Lab 반영 방식: latency histogram 또는 percentile bar로 빠른 요청과 느린 요청의 비율을 보여준다.

Codex 누락 방지 규칙: p95/p99 설명에는 latency distribution을 함께 연결한다.

### Resource Utilization

분류: 성능 지표 / 운영

정의: CPU, 메모리, DB connection, thread, file descriptor 같은 자원이 얼마나 사용 중인지 나타내는 지표다.

실무에서 중요한 이유: 응답 지연이 어떤 자원의 포화와 연결되는지 판단할 수 있다.

흔한 오해: CPU가 낮으면 서버에는 문제가 없다고 생각한다.

Visual Lab 반영 방식: Metric Card에 CPU, thread wait, DB pool, external connection pool을 분리해 표시한다.

Codex 누락 방지 규칙: 성능 장에는 latency와 최소 하나 이상의 자원 사용률을 함께 보여준다.

### Error Rate

분류: 성능 / 안정성 지표

정의: 전체 요청 중 실패한 요청의 비율이다.

실무에서 중요한 이유: 응답 시간이 빨라도 실패율이 높으면 서비스 품질은 나쁘다.

흔한 오해: 성능 테스트에서는 처리량과 응답 시간만 보면 된다고 생각한다.

Visual Lab 반영 방식: Load Test Curve와 Metric Card에서 error rate를 latency와 함께 표시한다.

Codex 누락 방지 규칙: 부하 증가 시 throughput, latency, error rate를 함께 비교한다.
