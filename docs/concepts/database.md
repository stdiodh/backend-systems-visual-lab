# Database Concepts

### 인덱스

분류: DB 성능

정의: DB가 특정 조건의 데이터를 더 빠르게 찾도록 돕는 자료 구조다.

실무에서 중요한 이유: 조회 트래픽이 많은 서비스에서 인덱스 부재는 성능 저하의 대표 원인이다.

흔한 오해: 모든 컬럼에 인덱스를 붙이면 좋다고 생각한다.

Visual Lab 반영 방식: 인덱스 없음/있음 비교 카드와 Query Path Diagram에 반영한다.

Codex 누락 방지 규칙: 인덱스 설명에는 쓰기 비용과 잘못된 인덱스의 한계를 함께 넣는다.

### 실행 계획

분류: DB 성능

정의: DB가 쿼리를 실행하기 위해 선택한 접근 경로와 처리 방식이다.

실무에서 중요한 이유: 쿼리가 왜 느린지 감이 아니라 근거로 확인할 수 있다.

흔한 오해: SQL 문장만 보면 DB 내부 동작을 알 수 있다고 생각한다.

Visual Lab 반영 방식: full scan, index scan, join 방식, rows examined를 경로로 표시한다.

Codex 누락 방지 규칙: DB 성능 개선 설명에는 SQL 문장과 내부 접근 경로를 분리해서 설명한다.

### N+1 문제

분류: DB 성능 / ORM

정의: 목록 1번 조회 후 각 항목의 연관 데이터를 가져오느라 추가 쿼리가 N번 발생하는 문제다.

실무에서 중요한 이유: 데이터가 적을 때는 티가 안 나지만 트래픽과 데이터가 늘면 급격히 느려진다.

흔한 오해: 코드에 명시적인 반복문이 없으면 쿼리도 한 번만 나간다고 생각한다.

Visual Lab 반영 방식: N+1 Timeline으로 첫 쿼리와 추가 쿼리를 시간 순서로 보여준다.

Codex 누락 방지 규칙: 문제 코드, 예상 쿼리 흐름, 대표 해결 방향을 함께 넣는다.

### 트랜잭션

분류: DB / 실패 처리

정의: 여러 DB 작업을 하나의 작업 단위로 묶어 모두 성공하거나 모두 실패하게 만드는 경계다.

실무에서 중요한 이유: 실패 시 데이터가 절반만 반영되는 문제를 막는다.

흔한 오해: 트랜잭션 안에서 외부 API를 호출해도 항상 안전하다고 생각한다.

Visual Lab 반영 방식: Transaction Boundary Panel에서 DB 작업과 외부 연동 경계를 분리해 보여준다.

Codex 누락 방지 규칙: 트랜잭션 설명에는 롤백과 외부 연동 주의 사항을 함께 넣는다.

### DB 분산 잠금

분류: DB / 동시성

정의: DB 테이블에 잠금 소유자와 만료 시간을 저장해 여러 서버 인스턴스 사이의 중복 실행을 막는 방식이다.

실무에서 중요한 이유: Redis 같은 별도 인프라 없이도 제한적인 분산 잠금 요구를 구현할 수 있다.

흔한 오해: row 하나를 넣으면 자동으로 모든 장애 상황이 해결된다고 생각한다.

Visual Lab 반영 방식: Lock Table Diagram, Acquire/Release Timeline, Timeout Scenario로 보여준다.

Codex 누락 방지 규칙: owner, expires_at, release 실패, transaction boundary를 반드시 포함한다.

### 복합 인덱스

분류: DB 성능

정의: 둘 이상의 컬럼을 묶어 만든 인덱스다.

실무에서 중요한 이유: 실제 조회 조건은 여러 컬럼을 함께 쓰는 경우가 많아 컬럼 순서와 조건 조합이 성능에 큰 영향을 준다.

흔한 오해: 단일 컬럼 인덱스를 여러 개 만들면 복합 조건도 항상 빠르다고 생각한다.

Visual Lab 반영 방식: Query Path Diagram에서 `status + created_at` 같은 조건 순서별 탐색 경로를 비교한다.

Codex 누락 방지 규칙: 복합 인덱스 설명에는 컬럼 순서와 정렬 조건을 포함한다.

### 선택도

분류: DB 성능

정의: 특정 조건으로 걸러지는 데이터 비율이다.

실무에서 중요한 이유: 선택도가 낮은 조건에 인덱스를 걸어도 기대만큼 성능이 좋아지지 않을 수 있다.

흔한 오해: 자주 검색하는 컬럼이면 무조건 좋은 인덱스 후보라고 생각한다.

Visual Lab 반영 방식: Index Comparison Card에서 `gender`, `status`, `created_at` 같은 예시를 선택도별로 비교한다.

Codex 누락 방지 규칙: 인덱스 설계 설명에는 조회 빈도뿐 아니라 선택도를 함께 다룬다.

### Keyset Pagination

분류: DB 조회 성능

정의: offset 대신 마지막으로 본 기준값을 이용해 다음 페이지를 조회하는 방식이다.

실무에서 중요한 이유: 깊은 페이지에서 offset scan 비용이 커지는 문제를 줄일 수 있다.

흔한 오해: 모든 페이징은 `offset`과 `limit`만으로 충분하다고 생각한다.

Visual Lab 반영 방식: Offset Pagination과 Keyset Pagination의 scan 범위를 나란히 보여준다.

Codex 누락 방지 규칙: 페이징 설명에는 정렬 기준, 중복/누락 가능성, UX 제약을 함께 포함한다.
