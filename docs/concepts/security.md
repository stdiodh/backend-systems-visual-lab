# Security Concepts

### 인증

분류: 보안

정의: 요청한 사용자가 누구인지 확인하는 과정이다.

실무에서 중요한 이유: 사용자를 식별하지 못하면 권한 판단도 할 수 없다.

흔한 오해: 로그인만 성공하면 모든 API 접근이 가능하다고 생각한다.

Visual Lab 반영 방식: AuthN/AuthZ Split Panel에서 인가와 분리한다.

Codex 누락 방지 규칙: 인증과 인가는 항상 분리해서 설명한다.

### 인가

분류: 보안

정의: 인증된 사용자가 특정 자원이나 기능을 사용할 권한이 있는지 판단하는 과정이다.

실무에서 중요한 이유: 로그인한 사용자라도 다른 사용자의 데이터나 관리자 기능에 접근하면 안 된다.

흔한 오해: 인증 성공과 인가 성공을 같은 것으로 본다.

Visual Lab 반영 방식: Resource Access Decision Card를 만든다.

Codex 누락 방지 규칙: 본인 글 수정, 관리자 기능 접근 같은 인가 예시를 넣는다.

### HMAC

분류: 보안 / 무결성 검증

정의: 공유 비밀키와 해시 함수를 이용해 메시지가 위변조되지 않았는지 확인하는 방식이다.

실무에서 중요한 이유: 외부 연동 요청이나 웹훅에서 데이터 위변조 여부를 검증할 수 있다.

흔한 오해: 암호화와 HMAC을 같은 개념으로 생각한다.

Visual Lab 반영 방식: Payload, Secret, Signature, Verify 흐름을 보여준다.

Codex 누락 방지 규칙: HMAC은 암호화가 아니라 무결성 검증으로 설명한다.

### 감사 로그

분류: 보안 / 운영

정의: 중요한 행위가 언제, 누가, 무엇을 대상으로, 어떤 결과로 수행됐는지 남기는 로그다.

실무에서 중요한 이유: 사고가 발생했을 때 추적과 책임 확인이 가능하다.

흔한 오해: 일반 서버 로그와 감사 로그를 같은 목적으로 본다.

Visual Lab 반영 방식: Audit Log Timeline으로 요청과 기록을 연결한다.

Codex 누락 방지 규칙: 로그인, 권한 변경, 중요 데이터 접근 예시를 포함한다.

### 데이터 노출 최소화

분류: 보안 / 개인정보

정의: 응답, 로그, 화면, 저장소에 불필요한 민감정보가 드러나지 않도록 줄이는 원칙이다.

실무에서 중요한 이유: 장애 로그나 API 응답 하나로도 개인정보 유출 사고가 생길 수 있다.

흔한 오해: DB 암호화만 하면 응답과 로그 노출은 문제가 아니라고 생각한다.

Visual Lab 반영 방식: Sensitive Data Masking Card와 unsafe/safe response 비교를 둔다.

Codex 누락 방지 규칙: 보안 장에는 로그 마스킹과 응답 필드 최소화를 함께 포함한다.
