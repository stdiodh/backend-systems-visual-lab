# Learning Flow

## 학습 설계 원칙

이 랩은 개념 설명으로 시작하지 않는다. 모든 장은 실무 문제 상황에서 시작하고, 학습자는 지표를 보며 원인 후보를 좁힌 뒤 해결 선택지를 비교한다.

## 표준 흐름

```text
Incident
-> Observations
-> Candidate Causes
-> Concepts
-> Decisions
-> Code or Structure
-> Checklist
```

## 화면 흐름

1. Lab Map에서 전체 장의 위치를 확인한다.
2. Incident Panel에서 문제 상황과 사용자 영향을 본다.
3. Metric Cards에서 관찰 지표를 본다.
4. Flow Explorer에서 요청, DB, 외부 연동, 메시지, 서버 자원을 따라간다.
5. Concept Cards에서 필요한 개념을 짧게 확인한다.
6. Decision Panel에서 선택지와 부작용을 비교한다.
7. Checklist에서 실무 누락 포인트를 확인한다.

## 반복 학습 루프

각 장은 이전 장의 문제를 다음 장으로 자연스럽게 넘긴다.

- 02장의 성능 진단은 03장의 DB 병목으로 이어진다.
- 03장의 트랜잭션 경계는 04장의 외부 연동 문제로 이어진다.
- 04장의 장애 전파는 05장의 비동기 실패 분리로 이어진다.
- 05장의 중복 처리와 순서 문제는 06장의 동시성으로 이어진다.
- 07장의 IO 모델은 04장과 05장의 외부 연동 처리와 연결된다.
- 08장의 보안은 모든 API와 외부 연동에 적용된다.
- 09장과 10장은 운영 중 원인 추적 능력을 보강한다.
- 11장은 앞 장들의 문제를 구조 선택으로 정리한다.

## 누락 방지

Codex는 장 구현 전 아래 질문에 답할 수 있어야 한다.

- 이 장의 시작 문제는 무엇인가?
- 학습자가 먼저 봐야 할 지표는 무엇인가?
- 흔한 오해는 무엇인가?
- 해결 선택지의 부작용은 무엇인가?
- 이 장이 이전 장이나 다음 장과 어떻게 연결되는가?
